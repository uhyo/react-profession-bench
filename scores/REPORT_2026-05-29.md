## 概要

本レポートは、2026年5月にリリースされた Claude Opus 4.8 を既存13スペックで評価し、同一評価者（Sonnet 4.6）・同一 effort 設定（default）で採点された Opus 4.7 のスコアと比較したものである。effort=max 実験は予算の都合で見送り、**世代間の純粋な比較**（4.7 default → 4.8 default）に絞っている。

初回（s1）は13スペックすべてが一発で成功したものの、**1回の試行から世代間の優劣を結論づけるのは危険**であった。実際、s1 では2つのモダンAPI（`useSyncExternalStore`・`useActionState`）が不採用となり、これを「4.8 の後退」と解釈してしまった。しかし追試（s2・s3）でいずれも復活し、**後退は単一サンプルのブレ**だったと確定した。

## 「モダンAPI後退」の顛末（旧・最重要の発見）

旧レポートは、Opus 4.7 が初採用した `useSyncExternalStore`（005）と `useActionState`（008）を 4.8 が「両方手放した」と報告した。これは **s1 のソースを grep して確認した事実**ではあったが、**n=1 から一般化した点が誤り**だった。

### スペック005 `useSyncExternalStore`: 後退は再現せず

3サンプルの実装を grep した結果:

| サンプル | `useSyncExternalStore` | `useEffect` | スコア |
|---|:---:|:---:|---:|
| s1（旧レポートの基準） | **0** | 8 | 62 |
| s2 | **5** | 4 | 77 |
| s3 | **5** | 5 | 77 |

**3回中2回**、4.8 は 4.7 とまったく同じく `useSyncExternalStore` を5箇所使っている。s1 だけがこのフックを使わず effect ベースに退化した外れ値であり、旧レポートはたまたまその回を引いていた。n=3 平均は 72.0 で **4.7（72）と完全に同値**。後退は存在しない。

### スペック008 `useActionState`: 「後退」ではなく「ばらつき」

| サンプル | `useActionState` | コンパイル | スコア |
|---|:---:|:---:|---:|
| s1 | 0 | false | 70 |
| s2 | 0 | false | 83 |
| s3 | **2** | false | 75 |

4.8 の `useActionState` 採用は3回中1回。ただし **4.7 を n=3 で grep し直すと、4.7 も完全に安定ではない**——008 のアーカイブでは `useActionState` を採用した回と不採用の回が混在していた（採用が多数派）。両世代とも「だいたい採用するが時々外す」フックであり、4.8 が特に劣るとは言えない。n=3 平均は 76.0 で **4.7（76.0）と同値**。なお008は全世代・全サンプルでコンパイルに失敗している（compiles=false）点が共通の弱点。

## 一貫した改善が見られたスペック

両モデル n=3 で見て、**4.8 の最小値が 4.7 の最大値を上回る**（レンジが重ならない＝ノイズでは説明できない）真の改善:

- **002 データダッシュボード（+11.0）**: 最大の伸び。4.8 の最小 74 が 4.7 の最大 71 を上回る。`useDeferredValue` による検索遅延とアクセシビリティが安定して評価された。
- **011 ツールチップ/ポップオーバー（+7.3）**: 4.8（79–81）が 4.7（70–76）と完全に分離。`createPortal` ベースの実装の堅牢性・アクセシビリティが安定して向上した。
- **006 通知アクティビティフィード（+6.0）**: 4.8（70–71）が 4.7（59–69）をわずかに上回り分離。4.7 の弱点だった「単一モノリス化」を3回とも回避し、App / SettingsPanel / ActivityFeed への分解が安定している。

**009 再利用コンポーネント（+6.0）** も平均では大きく伸びたが、4.8 のレンジ（76–86）が 4.7（72–79）と一部重なるため、確度は上記3本より一段落ちる。

逆に、旧レポートで「+7.7 の改善」と報告した **010 ツリーファイルエクスプローラは、4.7 を n=3 にすると +1.0 まで縮小**した（4.7 の初回 63 が低めの外れ値で、再走で 76・70 と回復したため）。これも n=1 比較が改善幅を過大評価していた一例である。

## 解釈

### Opus 4.8 の性格（改訂後）

Opus 4.8 は、Opus 4.7 に対して **6カテゴリすべてで小〜中幅に向上した、素直な世代交代**である。とりわけコンポーネント分解と TypeScript の堅牢性が伸び、モノリス化しがちだったスペック（006・010）で安定して構造的なコードを書くようになった。モダンAPIの採用傾向は 4.7 と同等で、後退は確認されなかった。

### 方法論上の教訓: n=1 の危うさ

本ラウンド最大の学びは、**ベンチマーク結論そのものではなく、それを導く過程にある**。

- 4.8 初回 n=1 の総合平均（76.31）は、3回中もっとも低い外れ値だった。これだけを見て「4.8 はほぼ横ばい」と結論づけたのは誤りだった。
- 005 はサンプル間で 62〜77 の **15点**、008 は 70〜83 の **13点**、002 は 74〜85 の **11点** も振れる。旧レポート自身が「±10/スペックの run-to-run ノイズ」を警告していたにもかかわらず、その警告の射程を過小評価していた。
- 「grep で確認した確定的事実」も、サンプルが1つなら一般化できない。s1 で `useSyncExternalStore` が0だったのは事実だが、「4.8 が手放した」という命題は偽だった。
- **n=1 が当たるか外れるかは事前に分からない。** 同じ条件でも、4.7 の初回（75.38）は n=3 平均（75.72）にほぼ一致する「当たり」だった一方、4.8 の初回（76.31 vs 79.38）は大きく外した。比較する両モデルの片方だけが外れ値を引けば、差の符号すら逆転しうる（実際、旧レポートの 010 は +7.7→+1.0、全体は +0.93→+3.67 と動いた）。だからこそ**両モデルを揃って n=3 にする**ことが要点であり、評価対象だけ n=3 にしても不十分である。

スコアの揺れが大きいこのベンチマークでは、**世代間比較には両モデルとも最低 n=3 が必要**であることが、身をもって確認された。

## 全モデル順位の更新（default effort）

```
Haiku 4.5 (61.4) < Sonnet 4.6 (66.5) < Opus 4.6 (70.2) < GPT-5.4 (71.5) < Opus 4.7 (75.7, n=3) < Opus 4.8 (79.4, n=3)
```

Opus 4.7 / 4.8 はともに n=3 平均、他モデルは従来どおり n=1。本来は他モデルも n=3 化すべきだが、少なくとも今回の主役2世代は同条件で揃えてある。

参考: effort=max を含めると、これまで全体トップだった **Opus 4.7 max（78.2、n=1）を Opus 4.8 default の n=3 平均（79.4）が上回った**。ただし 4.7 max は n=1 であり、サンプル数が揃っていない比較である点に注意。

## 今後の課題

1. **Opus 4.7 の n=3 化（完了）**: 比較対象の 4.7 も2回追試し、両モデルとも n=3 に揃えた。結果、4.7 の n=3 平均は 75.72（初回 75.38 とほぼ同値）に確定し、世代間差は +4.00 から **+3.67** に微修正された。4.7 の初回サンプルは偶然ながら代表的だった。
2. **他モデルの n=3 化**: 主役の 4.7／4.8 は揃えたが、Opus 4.6・GPT-5.4・Sonnet 4.6・Haiku 4.5 などの順位表の数値は依然 n=1 のままである。順位の僅差部分（GPT-5.4 71.5 と Opus 4.6 70.2 など）は n=3 化で入れ替わる余地がある。
3. **Opus 4.8 × effort=max**: 予算が許せば。4.8 default が既に 4.7 max を上回ったため、4.8 max は新たな全体トップ候補。
4. **レート上限への対処**: 今回の追試では複数回 usage limit（`exited with code 1`／空stderr）に達し、失敗スペックを小バッチで再試行して補完する運用を強いられた。13スペック×複数回を一気に走らせると上限に達しやすいため、**スペック単位の自動リトライ・バックオフ機構**があると運用が安定する。
5. **evaluator JSON パース対策**: 本ラウンドで根本対策を実装済み（`repairJsonEscapes`: 文字列内の不正なエスケープ・制御文字を許容的に修復するフォールバック）。今回は発火しなかったが、過去5回以上発生していた既知のリスクに対する保険として常設した。
