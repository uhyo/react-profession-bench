> **改訂履歴**: 本レポートは当初、Opus 4.8 を1回だけ走らせた結果（n=1）に基づき「モダンAPI採用の後退」を最重要の発見として報告していた。その後**同一条件で2回追試**して n=3 に拡張したところ、**この「後退」は再現しなかった**。初回サンプルが3回中もっとも低いスコアの外れ値だったことが判明したため、結論を全面的に改訂した。旧版の n=1 の数値は「初回サンプル（s1）」として表中に残してある。この顛末自体が、n=1 ベンチマークの危うさを示す重要な事例である（後述）。

## 概要

本レポートは、2026年5月にリリースされた Claude Opus 4.8 を既存13スペックで評価し、同一評価者（Sonnet 4.6）・同一 effort 設定（default）で採点された Opus 4.7 のスコアと比較したものである。effort=max 実験は予算の都合で見送り、**世代間の純粋な比較**（4.7 default → 4.8 default）に絞っている。

初回（s1）は13スペックすべてが一発で成功したものの、**1回の試行から世代間の優劣を結論づけるのは危険**であった。実際、s1 では2つのモダンAPI（`useSyncExternalStore`・`useActionState`）が不採用となり、これを「4.8 の後退」と解釈してしまった。しかし追試（s2・s3）でいずれも復活し、**後退は単一サンプルのブレ**だったと確定した。

## 「モダンAPI後退」の顛末（旧・最重要の発見）

旧レポートは、Opus 4.7 が初採用した `useSyncExternalStore`（005）と `useActionState`（008）を 4.8 が「両方手放した」と報告した。これは **s1 のソースを grep して確認した事実**ではあったが、**n=1 から一般化した点が誤り**だった。

### スペック005 `useSyncExternalStore`: 後退は再現せず

3サンプルの実装を grep した結果:

| サンプル | `useSyncExternalStore` | `useEffect` | スコア |
|---|:---:|:---:|---:|
| s1（旧レポートの基準） | **0** | 8 | 62 |
| s2 | **5** | 4 | 77 |
| s3 | **5** | 5 | 77 |

**3回中2回**、4.8 は 4.7 とまったく同じく `useSyncExternalStore` を5箇所使っている。s1 だけがこのフックを使わず effect ベースに退化した外れ値であり、旧レポートはたまたまその回を引いていた。n=3 平均は 72.0 で **4.7（72）と完全に同値**。後退は存在しない。

### スペック008 `useActionState`: 「後退」ではなく「ばらつき」

| サンプル | `useActionState` | コンパイル | スコア |
|---|:---:|:---:|---:|
| s1 | 0 | false | 70 |
| s2 | 0 | false | 83 |
| s3 | **2** | false | 75 |

`useActionState` の採用は3回中1回。確かに不安定だが、4.7 側も n=1（採用1回）でしか観測していない以上、「4.8 が劣る」とは言えない。n=3 平均は 76.0 で、これも **4.7（76）と同値**。なお008は全世代・全サンプルでコンパイルに失敗している（compiles=false）点は共通の弱点。

## 一貫した改善が見られたスペック

レンジが狭く（=3サンプルで安定して高い）、ノイズでは説明できない真の改善:

- **002 データダッシュボード（+12.3、レンジ74–85）**: 最大の伸び。`useDeferredValue` による検索遅延とアクセシビリティが安定して評価された。
- **006 通知アクティビティフィード（+11.3、レンジ70–71）**: 4.7 default の弱点だった「単一モノリス化」を3回とも回避。App / SettingsPanel / ActivityFeed への分解が安定している。レンジがわずか1点で、改善の確度が高い。
- **004 ユーザープロフィール閲覧（+7.7）**・**010 ツリーファイルエクスプローラ（+7.7）**: いずれもコンポーネント分解品質の底上げ。010 は 4.7 で「303行のモノリス」と酷評されたスペックで、3回とも状態設計・Effect衛生が回復した。

## 解釈

### Opus 4.8 の性格（改訂後）

Opus 4.8 は、Opus 4.7 に対して **6カテゴリすべてで小〜中幅に向上した、素直な世代交代**である。とりわけコンポーネント分解と TypeScript の堅牢性が伸び、モノリス化しがちだったスペック（006・010）で安定して構造的なコードを書くようになった。モダンAPIの採用傾向は 4.7 と同等で、後退は確認されなかった。

### 方法論上の教訓: n=1 の危うさ

本ラウンド最大の学びは、**ベンチマーク結論そのものではなく、それを導く過程にある**。

- 初回 n=1 の総合平均（76.31）は、3回中もっとも低い外れ値だった。これだけを見て「4.8 はほぼ横ばい」と結論づけたのは誤りだった。
- 005 はサンプル間で 62〜77 の **15点**、008 は 70〜83 の **13点**、002 は 74〜85 の **11点** も振れる。旧レポート自身が「±10/スペックの run-to-run ノイズ」を警告していたにもかかわらず、その警告の射程を過小評価していた。
- 「grep で確認した確定的事実」も、サンプルが1つなら一般化できない。s1 で `useSyncExternalStore` が0だったのは事実だが、「4.8 が手放した」という命題は偽だった。

スコアの揺れが大きいこのベンチマークでは、**世代間比較には最低でも n=3 が必要**であることが、身をもって確認された。

## 今後の課題

1. **Opus 4.7 の n=3 化（最優先）**: 今回 4.8 を n=3 にした一方、比較対象の 4.7 は依然 n=1 のままである。4.8 で得た教訓そのものから、**4.7 も2回追試して n=3 に揃えない限り、+4.00 という差の正確な大きさは確定できない**。4.7 の単一サンプル（75.38）が高め・低めのどちらの外れ値かは未検証。
2. **Opus 4.8 × effort=max**: 予算が許せば。4.8 default が既に 4.7 max を上回ったため、4.8 max は新たな全体トップ候補。
3. **レート上限への対処**: s3 では011–013が usage limit（`exited with code 1`／空stderr）で失敗し、別枠で再試行して補完した。13スペック×複数回を一気に走らせると上限に達しやすいため、スペック単位の自動リトライ・バックオフ機構があると運用が安定する。
4. **evaluator JSON パース対策**: 本ラウンドで根本対策を実装済み（`repairJsonEscapes`: 文字列内の不正なエスケープ・制御文字を許容的に修復するフォールバック）。今回は発火しなかったが、過去5回以上発生していた既知のリスクに対する保険として常設した。
