# React Profession Bench 第8回レポート: Claude Fable 5 — 新世代は max-effort に届くか

**実施日**: 2026年6月10日
**評価対象**: Claude **Fable 5**（新規・新世代フラッグシップ、default effort、n=3）
**比較対象**: Claude Opus 4.8 **default**（第6回、n=3）／Claude Opus 4.8 **max**（第7回、n=3）
**評価者モデル**: Claude Sonnet 4.6（過去全レポートと同一）
**スペック数**: 13
**サンプル数**: 3条件すべて **3回**（n=3、計117スペック分）。同一評価者・同一スペックで、モデル（と effort）だけを変えた対照である。

## 概要

2026年6月にリリースされた新世代フラッグシップ **Claude Fable 5** を、既存13スペックで n=3 評価した。問いは二つある——**(1) Fable 5 は現行最強の Opus 4.8 を超えるのか。(2) 「世代を上げる」ことは「effort を上げる」ことに勝てるのか。**

第7回で、Opus 4.8 は effort=max にすると default から +4.64 伸び、84.03 でベンチ単独トップに立った。今回の Fable 5 は **default effort** での測定である（max 相当の高 effort 版は本稿では測っていない）。したがって本来のフェアな比較軸は「Fable 5 default 対 Opus 4.8 default」だが、ベンチ全体での位置づけを見るため Opus 4.8 max とも並べる。

結論を先に述べる。**Fable 5 は Opus 4.8 default を確かに上回るが、その幅は小さく（+1.95）、Opus 4.8 max には届かない（−2.70）。** 新世代は1ランク前進したが、その前進幅は「同じ世代で effort を max に上げる」効果より小さい。

- 13スペック平均は **Opus 4.8 default 79.38 → Fable 5 81.33（Δ +1.95）**。
- 一方 **Opus 4.8 max（84.03）には 2.70 及ばない**。Fable 5 は「default 超え・max 未満」の中間に着地した。
- 全モデル順位では **Opus 4.8 default < Fable 5 < Opus 4.8 max** の位置に入る。

## カテゴリ別平均の変化（n=3、各39スペック分）

| カテゴリ | 4.8 def | **Fable 5** | 4.8 max |

| TypeScript品質 | 4.82 | **4.79** | 4.82 |
| パフォーマンス意識 | 3.67 | **3.79** | 4.00 |
| アクセシビリティ | 3.59 | **3.74** | 3.74 |

このカテゴリ表が Fable 5 の性格を最もよく表している。**Fable 5 はほぼ全カテゴリで「default と max のあいだ」に収まる。** ただし二つの例外がある。

- **アクセシビリティ（3.74）は max と同値**。Fable 5 はセマンティクスとアクセシビリティを、追加 effort なしで max-effort 相当まで作り込む。新世代の地力が最も素直に出た領域である。
- **TypeScript品質（4.79）だけは default をわずかに下回る**（4.82 → 4.79）。第7回で示したとおりこの領域は default で既に天井（4.82/5.00）に達しており、世代交代でも伸びない。むしろ Fable 5 は微減で、ここは「世代」でも「effort」でも動かない上限領域だと再確認された。
- **max との差はほぼ全部「コンポーネント設計」に集約**している。Fable 5 3.77 に対し max 4.21（−0.44）。第7回で max-effort の伸びが集中したのもこのカテゴリだった。つまり **Fable 5 に足りないのは「素の能力」ではなく「コンポーネント分解にかける思考量」**であり、これは effort で買える種類の差である——だから default 同士なら Fable 5 が上回り、相手が max になると逆転する。

## Fable 5 が「効く」スペック・「効かない」スペック

default とレンジが重ならない真の差は3本のみ——勝ち2本（010・013）、負け1本（006）。+1.95 という控えめな総合差に見合う、限定的な分布である。

**伸びた:**
- **013 設定画面Undo/Redo（+10.0、レンジ分離）**: 最大の伸び。Fable 5（90–94）が default（80–84）と完全分離し、**max（85.3）すら +6.7 上回る**。本スペックは Fable 5 が唯一 max を明確に超えた一本で、Undo/Redo の履歴設計という状態管理中心の課題で新世代の地力が出た。
- **010 ツリーファイルエクスプローラ（+7.0、レンジ分離）**: モノリス化しやすい題材で安定して分解できている。ただし max（80.7）には届かない。
- **005（+8.3）・002（+7.7）** も大きく伸びたが、いずれも default のレンジ上端と接しており「完全分離」には一歩届かない。

**伸びなかった・後退した:**
- **009 再利用コンポーネント（−7.3）**: 最大の後退。状態設計（5.00）・アクセシビリティ（4.00）は default と同等だが、**コンポーネント設計・TypeScript品質・パフォーマンス意識がそろって −0.67**。再利用可能な API 設計という「丁寧な分解と型付けが効く」課題で、Fable 5 default は Opus 4.8 default に一段譲った。3サンプルとも 75–76 と低位安定しており、ブレではなく傾向である。
- **006 通知フィード（−2.7、レンジ分離）**: 小幅だが明確な後退。状態設計が落ちた（4.67 → 4.00）一方アクセシビリティは上がっており（3.00 → 4.00）、得意・不得意の出方が default と微妙に異なる。
- **011 ツールチップ（−10.0 vs max）**: default とはほぼ同等（−1.3）だが、max（89.0）との差が全スペック中最大。`createPortal` ベースの作り込みは、ここでも effort 差がそのまま出ている。

```
Haiku 4.5 (61.4) < Sonnet 4.6 (66.5) < Opus 4.6 (70.2) < GPT-5.4 (71.5)
  < Opus 4.7 (75.7, n=3) < Opus 4.8 default (79.4, n=3)
```

**Fable 5（81.33）は Opus 4.8 default と max のあいだに単独で入る。** 新世代フラッグシップは現行 default 最強を超えたが、その差は +1.95 にとどまり、**同一世代を max effort に振った Opus 4.8（+4.64）には及ばない**。第6回で測った世代交代1回分（4.7→4.8 default の +3.67）と比べても、今回の 4.8→Fable 5 の +1.95 は小さい。**「Fable 5 は Opus 4.8 の正常進化だが、その進化幅は近年の世代交代より控えめ」**というのが13スペックの示す像である。

## コスト効率の注記

Fable 5 は **Opus の約2倍の使用量を消費する**（本実験中の実測感としても、1サンプルが1回の5時間 usage window に収まらず、s2・s3 で各1回ずつ usage limit に到達した。Opus 4.8 default の n=3 では限界到達はもっと稀だった）。

品質差とコストを重ねると、本ベンチに限れば Fable 5 default の費用対効果は厳しい:

- **対 Opus 4.8 default**: 約2倍のコストで +1.95。
- **対 Opus 4.8 max**: Fable 5 の方が高コストにもかかわらずスコアは −2.70 下。

つまり「同じ予算を積むなら、Fable 5 default に払うより Opus 4.8 を max に上げる方が、この13スペックでは費用対効果が良い」可能性が高い。ただし——**これは default effort の Fable 5 に限った話**である。Fable 5 を高 effort で走らせれば像は変わりうるし（max カテゴリ差がコンポーネント設計に集中している以上、effort で埋まる余地は大きい）、コストの厳密計測（トークン・実行時間）も本稿は行っていない。最終的な費用対効果判断は次の課題に委ねる。

## 運用面: probe-to-wake の初実戦

第7回までに整備した中断耐性機構に、本実験で **probe-to-wake**（usage window の早期復帰検知）を追加し、初めて実戦投入した。

第7回までの usage-limit 待機は、リセット時刻が取得できない（空出力で落ちる）場合に **固定で約5時間5分** 眠る方式だった。だが Anthropic の5時間 window はローリングで、しばしばもっと早く回復する。実際 s1 の初回 limit では、固定待機が真のリセットより **約80分も長く**眠ってしまい、手動で起こす必要があった。

そこで待機ロジックを「固定スリープ」から「**ハードキャップ付きポーリング**」に変えた。算出時刻（プロバイダ提示 or 5h05m フォールバック）は**上限**としてのみ使い、5分ごとに軽量プローブを撃って**最初に成功した瞬間に再開**する。本実験での実績:

- **s2 の limit（07:00 UTC 到達）**: ハードキャップ 12:05 に対し、**10:52 にプローブ成功で早期再開**（実際の window 回復は約3時間52分後）。固定待機なら被っていた約73分の空転を回収。
- **s3 の limit（12:00 UTC 到達）**: 同じく **15:52 に早期再開**（約3時間52分後）、約73分を回収。

2回とも真のローリング window は約3h52m で回復しており、**5h05m の固定フォールバックは実態より1時間以上保守的**だったことが裏づけられた。probe-to-wake により、**リセット時刻を一切パースできなくても、空転は最大でもプローブ間隔（5分）以内**に縮む。第7回で手動介入が要った over-wait は、今回は無人で自動回収された。

## 今後の課題

1. **Fable 5 の高 effort 化**: 本稿は Fable 5 **default** のみ。max との差がコンポーネント設計に集中している以上、Fable 5 を高 effort で走らせれば Opus 4.8 max を超える可能性がある。世代と effort の二軸を Fable 5 でも揃えるのが次の筋。
2. **コスト計測**: 「約2倍」は体感・限界到達頻度ベースにとどまる。トークン消費・実行時間を実測し、費用対効果を定量化すべきである。
3. **009 再利用コンポーネントの後退の精査**: 新世代が API 設計スペックで一段下げたのは示唆的。出力を読み、何が分解・型付けで劣ったのかを定性分析する価値がある。
4. **他モデルの n=3 化（継続課題）**: 順位表の Opus 4.6・GPT-5.4・Sonnet 4.6・Haiku 4.5 は依然 n=1。
